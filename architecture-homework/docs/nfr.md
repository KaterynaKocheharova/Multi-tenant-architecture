# Нефункціональні вимоги (NFR)

## RESILIENCY

### Стійкість до відмов

Система має бути готова працювати, навіть якщо допоміжні сервіси повертають помилки.
Наприклад:

Redis недоступний
❌ Помилка 500
✅ Система запитує дані з БД

Сервіс email недоступний
❌ Помилка 500
✅ Email додається в чергу, повторюється пізніше або використовується резервний сервіс

Один API-сервер упав
❌ Неможливість користуватися додатком
✅ Балансувальник навантаження відправляє запити на інші 3 сервери

### Обробка помилок

- **Логіка повторних спроб** з експоненційною затримкою:
  ```
  Спроба 1: одразу
  Спроба 2: 1 сек
  Спроба 3: 2 сек
  Спроба 4: 4 сек
  Спроба 5: 8 сек
  ```

### Резервне копіювання БД та аварійне відновлення

- **Частота резервного копіювання**: Щодня (авторезервне копіювання)
- **Термін зберігання**: Мінімум 30 днів

## SCALABILITY

### Горизонтальне масштабування

- **API-сервери**: додавати нові інстанси
- **Балансування навантаження**: щоб централізовано перенаправляти запити на найбільш підходящий сервер

### База даних

- **Основний шлях (поточний)**: спільний PostgreSQL + RLS + партиціювання таблиць через `schoolId`.
- **Умовний майбутній шлях**: шардінг тенантів (school -> окрема БД/сервер) лише у випадку сильного прояву проблеми "шумного сусіда"

## OBSERVABILITY

### Рівні логування

INFO: Загальний хід (вхід користувача, старт API-запиту)
WARN: Неочікувана, але відновлена ситуація (retry #2, мало місця на диску)
ERROR: Неочікувана ситуація (API-запит провалився, доступ заборонено)
FATAL: Система не може продовжувати роботу (база даних недоступна, нестача пам'яті)

### Метрики для відстеження

- **Бізнес-метрики**:
  - Нові реєстрації за день
  - Активні користувачі на школу
  - Успішність генерації звітів
  - Рівень участі в подіях
- **Технічні метрики**:
  - Час відповіді API
  - Частка помилок по ендпоїнтах
  - Час виконання запитів до БД
  - Час обробки задач у черзі
