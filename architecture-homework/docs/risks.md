# РИЗИКИ

## 1. Ризик noisy neighbor

Один tenant з високим навантаженням може погіршити продуктивність для всіх інших tenant-ів.

**Можливе рішення:**

1. Упровадити норму часу, за яку кожен користувач має отримувати відповідь від бази, наприклад, 100-300 мс.
2. Використовувати систему моніторингу, що допоможе виявити, у межах якого тенанта відбувається найбільше запитів.
3. Застосувати шардінг і винести дані що стосуються найактивнішого тенанта окремо.

## 2. Перевантаження сервера чисельними важкими запитами у гарячі періоди.

Якщо CPU-навантаження рівномірно зростає у межах усіх тенантів (наприклад, у період генерації репортів укінці року, коли всі вчителі різко почнуть їх створювати та редагувати), додатково можна:

1. Виконувати важкі обчислення у worker thread — це знизить навантаження на Node.js event loop
2. Можна використовувати декілька інстансів та лоуд беленсер.
3. Можна використовувати crone jobs які будуть заздалегідь створювати репорти (проте у спеціальній черзі не всі за раз) і додавати їх у редіс для швидкого читання.

## 3. Подовжений час реквесту від перевірки deny_list (куди зберігаємо access_token jwt після вилогіну для ротації).

**Можливі рішення:**

- Вимірювати час, і у випадку сповільнення запитів тимчасово її не виконувати.
- Кешувати токени у редіс, а крон джоба ранитиметься щодня і видалятиме всі записи по deny_list старші за 15 хвилин.
- Лишити перевірку лише для критичних ендпоїнтів.

## 4. Ризик викраденого JWT

Це може бути зроблено через ** XSS:**

Cross-Site Scripting — атака, коли зловмисник впроваджує шкідливий JavaScript на сторінку (наприклад, через незахищений input). Навіть якщо `access_token` зберігається in-memory, впроваджений скрипт може прочитати його зі стану застосунку або перехопити під час API-запиту.

Якщо access token буде викрадено, зловмисник зможе виконувати запити від імені користувача до закінчення строку дії токена.

**Можливі рішення**:

1. Версель кофігурації — [Security Headers Config](security.md#14-vercel-security-headers-config).
   Одна з цих конфігурацій забороняє сайту виконувати сторонні скрипти.
2. Робити короткий TTL для access token (наприклад, 5-15 хв).
3. Refresh token тримати тільки в `HttpOnly` + `Secure` cookie, не в localStorage.
4. Додати ротацію refresh token при кожному `POST /auth/refresh`.
5. Вести denylist для `jti` і перевіряти його для критичних endpoint-ів.
6. Логувати аномалії сесій (новий IP/UA, різкі паралельні запити) і ревокати сесію.
7. Мінімізувати дані в JWT (не класти чутливі поля), використовувати тільки потрібні claims.
8. Зберігати ексіс токен як кукі також (httpOnly, sameSite=Lax).

## 5. Ризик випадкового прямого читання таблиці user в API-коді

Коли в проєкт додається новий endpoint або сервіс (наприклад, `GET /events/:id/participants`), розробник може природно використати Prisma-модель `User` і зробити join. Якщо захист на рівні бази не встановлений, запит поверне всі поля: `passwordHash`, `email`, `isActive` тощо — і потрапить у відповідь або лог. Ризик зростає з часом: що більша кодобаза, то більша ймовірність такого випадкового витоку.

**Handling — захист на трьох рівнях:**

**Рівень 1: DB — REVOKE (найнадійніший catch-all)**

Встановлюється один раз. Будь-який прямий SELECT до `user` з `app_api_role` кине `permission denied` на рівні бази — незалежно від того, що написав розробник у коді.

```sql
REVOKE SELECT ON "user" FROM app_api_role;
GRANT SELECT ON user_public_view TO app_api_role;
```

**Рівень 2: Prisma — окрема модель для view (захист на рівні TypeScript)**

У `schema.prisma` оголошуємо view як окрему модель з лише безпечними полями. TypeScript не дозволить дістатись до `passwordHash` через цю модель — поля там просто немає.

```prisma
/// Безпечний публічний вид — лише id та fullName.
/// Не містить passwordHash, email, isActive.
view UserPublicView {
  id       String @id
  fullName String

  @@map("user_public_view")
}
```

Тоді в сервісі:

```typescript
// ПРАВИЛЬНО — тільки публічні поля, TypeScript не дасть дістатись до passwordHash
const participants = await prisma.userPublicView.findMany({
  where: { id: { in: participantIds } },
});

// НЕПРАВИЛЬНО — Prisma кине DB-помилку (REVOKE спрацює),
// і TypeScript підкаже, що User-модель не призначена для цього контексту
const participants = await prisma.user.findMany({ ... });
```

**Рівень 3: Code review**

- Не імпортувати `prisma.user.*` у модулях, пов'язаних з events або participants.
- У PR перевіряти, що запити до учасників завжди йдуть через `userPublicView`.

> Рівні 1 і 2 разом дають подвійний захист: база блокує помилковий запит навіть якщо TypeScript-типи було обійдено (наприклад, через raw query). Рівень 3 — превентивний (до merge).

## 2. Ризик витоку міжtenant-даних через GLOBAL events

Якщо, наприклад, журі глобального івенту з одної школи отримує деякі дані по учаснику змагання іншої школи,
бути доступними при запиті.
Або, інша ситуація, коли всі юзери бачать івенти, проте під час фетчінгу івенту було зафетчено також і організватора разом із його приватними полями.

Таблиця `user` не має поля `schoolId`, тому стандартну RLS-ізоляцію за тенантом до неї застосувати не можна — користувачі є "глобальними" сутностями, прив'язаними до тенантів лише через `Membership`. Це означає, що ризик витоку через `join` є цілком реальним і потребує окремого захисту.

Окремий ризик тут у тому, що RLS не "успадковується" автоматично через `join`. Якщо запит читає `event_participation` і одночасно `user`, то PostgreSQL окремо застосовує policy до кожної таблиці. Тобто сам факт, що рядок `event_participation` доступний, не означає, що доступ до `user` також безпечний. Якщо policy на `user` написана занадто широко, або API-роль взагалі має прямий `SELECT` на `user`, то через global event можна підтягнути приватні дані користувача.

**Можливе рішення:**

> **Перевірено по [data-model.md](data-model.md):** поле `fullName` знаходиться безпосередньо на таблиці `user` (`fullName text not null`). `TeacherDetails` та `StudentDetails` містять лише `musicalInstrument`/`classNumber` і не потрібні для списку учасників події. Тому `user_public_view` з `id, full_name` є коректним рішенням.

1. Не дозволяти жодному read-path для `GLOBAL events` читати сиру таблицю `user`.

2. Створити SQL **view** `user_public_view` лише з безпечними колонками.
   `security_barrier = true` — запобігає оптимізаційним підстановкам планувальника запитів (predicate pushdown), які могли б обійти фільтри view і витекти дані через timing-атаки.

```sql
CREATE VIEW user_public_view
WITH (security_barrier = true) AS
SELECT
  id,
  full_name
FROM "user";
```

```sql
-- безпечно: поверне лише id і full_name, решта полів фізично недоступна
SELECT * FROM user_public_view WHERE id = $1;
```

4. **Щоб це не можна було обійти через join/relation** (наприклад `event_participation -> user`), блокуємо доступ до сирої таблиці і даємо доступ тільки до safe-view:

`app_api_role` — PostgreSQL-роль рівня бази даних, під якою NestJS-сервер підключається до PostgreSQL. Не плутати з ролями застосунку (`SCHOOL_ADMIN`, `TEACHER` тощо) — це саме DB-роль. Відкликаючи у неї прямий `SELECT` на `user`, ми гарантуємо, що жодна частина коду не зможе прочитати повний рядок користувача — навіть через raw SQL у сервісах.

```sql
-- роль API не має права читати базову таблицю user
REVOKE SELECT ON "user" FROM app_api_role;

-- але має право читати лише безпечний view
GRANT SELECT ON user_public_view TO app_api_role;
```

Приклад безпечного join для `participants`:

```sql
-- ep = event_participation (alias), u = user_public_view (alias)
SELECT
  ep.event_id,
  ep.participant_user_id,
  u.id,
  u.full_name
FROM event_participation ep
JOIN user_public_view u ON u.id = ep.participant_user_id
WHERE ep.event_id = $1;
```

4. Для найбільшого контролю можна зробити ще `global_event_public_view`, де organizer уже приєднаний у безпечному вигляді, щоб клієнт і прикладний код узагалі не будували `join` до `user` самостійно.

### Питання про RLS та views

**Що таке view?**

View — це збережений SQL-запит із власним іменем. Коли ти пишеш `SELECT * FROM user_public_view`, PostgreSQL фактично виконує підзапит, що лежить всередині view. Фізично нових даних немає — view це просто "лінза" над таблицею. Головна перевага тут: код фізично не може прочитати `passwordHash` або `email`, навіть якщо напише `SELECT *` — цих колонок у view просто немає.

**Якщо я напишу RLS policy безпосередньо на `user` — чи потрібні views взагалі?**

Не обов'язково, але це вирішують різні проблеми:

| Підхід                      | Що контролює                       | Коли підходить                                                                     |
| --------------------------- | ---------------------------------- | ---------------------------------------------------------------------------------- |
| **RLS policy на `user`**    | Які **рядки** (users) можна бачити | Є чіткий критерій ізоляції на рівні рядків, наприклад, тенантний `schoolId`        |
| **View (user_public_view)** | Які **колонки** можна бачити       | Таблиця глобальна — рядки не можна ізолювати за тенантом, але треба приховати поля |

Для `user` конкретно: таблиця **глобальна** (немає `schoolId`), тому row-level ізоляцію за тенантом не можна застосувати стандартним шаблоном `tenant_id = current_setting(...)`. RLS policy тут довелося б будувати через підзапити ("можна бачити лише users, що беруть участь у тих самих events, що і поточний тенант"), — це нетривіально і важко підтримувати. View + REVOKE простіше і більш стійке до помилок.

Але якби у `user` був `schoolId`, можна було б написати RLS policy — і тоді views були б не обов'язкові. Обидва підходи можна й комбінувати.

**Як отримати user з даними профілю (TeacherDetails / StudentDetails) якщо є REVOKE на `user`?**

View не заважає цьому — ти так само можеш зробити JOIN до `teacher_details` або `student_details`, просто замість сирої таблиці `user` використовуєш `user_public_view`. Поле `id` у view є, а саме через нього `teacher_details.user_id` і `student_details.user_id` і з'єднуються. `TeacherDetails`/`StudentDetails` не містять чутливих полів і REVOKE на них не поширюється.

```sql
-- отримати учасників події з їхніми студентськими даними
SELECT
  ep.event_id,
  u.id,
  u.full_name,
  sd.musical_instrument,
  sd.class_number
FROM event_participation ep
JOIN user_public_view u      ON u.id = ep.participant_user_id
LEFT JOIN student_details sd ON sd.user_id = u.id
WHERE ep.event_id = $1;
```

```sql
-- або з даними викладача
SELECT
  u.id,
  u.full_name,
  td.musical_instrument
FROM event_participation ep
JOIN user_public_view u      ON u.id = ep.participant_user_id
LEFT JOIN teacher_details td ON td.user_id = u.id
WHERE ep.event_id = $1;
```

Якщо тобі потрібні обидва профілі одночасно (наприклад, учасник може бути і вчителем, і учнем), обидва LEFT JOIN можна додати в одному запиті.

`LEFT JOIN` тут — навмисний: не кожен user має запис у `student_details` (наприклад, `JURY` або `TEACHER` не є студентами), тому `INNER JOIN` відкинув би їх із результатів.

Так. PostgreSQL застосовує RLS-policy **кожної таблиці незалежно**, навіть якщо таблиця читається через JOIN. Те, що основний запит (`event_participation`) пройшов свою policy — не відкриває автоматично `user` у JOIN. PostgreSQL окремо перевіряє policy `user` для кожного рядка, до якого намагається дотягнутись через join.

**Що відбувається, якщо основний запит пройшов policy, а joined-таблиця — ні?**

PostgreSQL **не кидає помилку** і не повертає error. Він тихо поводиться так, ніби рядок не існує:

- `INNER JOIN user` → весь рядок `event_participation` **не повертається** (ніби user-а немає взагалі).
- `LEFT JOIN user` → рядок `event_participation` **повертається**, але всі колонки з `user` будуть `NULL`.

`LEFT JOIN` особливо небезпечний: ти можеш вирішити, що user-поля просто відсутні або не заповнені — а не те, що тебе заблокувала policy. Тому важливо знати свої policies і покривати цей сценарій тестами.

## 6. Ризик помилки в RLS policy (витік або блокування даних між тенантами)

Неправильно написана RLS-policy може або **витекти** дані (занадто дозвільна — повертає рядки інших тенантів), або **заблокувати** легітимний доступ (занадто обмежена — повертає порожній результат там, де мали бути дані). Обидва варіанти небезпечні: перший порушує privacy, другий може непомітно поламати функціональність.

**Можливе рішення:**

1. Тестувати RLS-policy з тестовими користувачами від різних тенантів (перевіряти і витік, і блокування).
2. Використовувати `FORCE ROW LEVEL SECURITY`, щоб навіть власник таблиці підпадав під policy.
3. Покривати ізоляцію тенантів інтеграційними тестами: "тенант A не бачить даних тенанта B".
4. Переглянути приклади у [rls-details.md](some-low-level-files/rls-details.md).
