# РИЗИКИ

## 1. Ризик noisy neighbor у database

Один tenant з високим навантаженням може погіршити продуктивність для всіх інших tenant-ів.

**Можливе рішення:**

1. Упровадити норму часу, за яку кожен користувач має отримувати відповідь від бази, наприклад, 100-300 мс.
2. Використовувати систему моніторингу, що виявлятиме порушення від норми, що допоможе виявити, який у межах якого тенанта відбувається найбільше запитів.
3. Використати окрему базу даних для даного тенанта або здійснити шардінг бази даних.

Або: якщо проблема не база даних, то можна здійснити горизонтальне розширення, придбати більше серверів і додати лоуд балансер.

## 2. Ризик витоку міжtenant-метаданих через GLOBAL events

Якщо, наприклад, журі з одної школи отримує деякі дані по учаснику змагання іншої школи, то існує ризик витоку даних, які не мали
бути доступними при запиті.
Або, інша ситуація, коли всі юзери бачать івенти, проте під час фетчінгу івенту було зафетчено також і організватора разом із його приватними полями.

Окремий ризик тут у тому, що RLS не "успадковується" автоматично через `join`. Якщо запит читає `event_participation` і одночасно `user`, то PostgreSQL окремо застосовує policy до кожної таблиці. Тобто сам факт, що рядок `event_participation` доступний, не означає, що доступ до `user` також безпечний. Якщо policy на `user` написана занадто широко, або API-роль взагалі має прямий `SELECT` на `user`, то через global event можна підтягнути приватні дані користувача.

**Можливе рішення:**

1. Не дозволяти жодному read-path для `GLOBAL events` читати сиру таблицю `user`.

2. Створити SQL **view** `user_public_view` лише з безпечними колонками.

```sql
CREATE VIEW user_public_view
WITH (security_barrier = true) AS
SELECT
  id,
  full_name
FROM "user";
```

```sql
-- безпечно: поверне лише id і full_name, решта полів фізично недоступна
SELECT * FROM user_public_view WHERE id = $1;
```

3. **Щоб це не можна було обійти через join/relation** (наприклад `event_participation -> user`), блокуємо доступ до сирої таблиці і даємо доступ тільки до safe-view:

```sql
-- роль API не має права читати базову таблицю user
REVOKE SELECT ON "user" FROM app_api_role;

-- але має право читати лише безпечний view
GRANT SELECT ON user_public_view TO app_api_role;
```

Приклад безпечного join для `participants`:

```sql
SELECT
  ep.event_id,
  ep.participant_user_id,
  upv.id,
  upv.full_name
FROM event_participation ep
JOIN user_public_view upv ON upv.id = ep.participant_user_id
WHERE ep.event_id = $1;
```

4. Для найбільшого контролю можна зробити ще `global_event_public_view`, де organizer уже приєднаний у безпечному вигляді, щоб клієнт і прикладний код узагалі не будували `join` до `user` самостійно.

5. Використовувати тести, які перевіряють обидва сценарії:

- без tenant context запит до сирого `user` повертає `deny` або помилку;
- запит до `user_public_view` для organizer/participant глобального івенту повертає лише дозволені поля.

6. Покладатися не лише на RLS. RLS тут є додатковим шаром захисту, але основний контроль витоку приватних полів має забезпечуватися через safe-view або окрему public projection.

## 3. Ризик складності політик бази даних для забезпечення ізоляції даних.

Дуже складно зрозуміти ці політики, треба буде сетапити дуже багато перед розробкою.

**Можливе рішення**

Долучити експертів які це зроблять якісно і швидко або проконсультуватися з ними щодо доречності використання політи. Можливо, можна без них.

## 4. Додатковий час до реквесту від перевірки чи джейсон веб токен є у денай списку чи ні.

**Можливе рішення:**

- Вимірювати час, і у випадку сповільнення запитів тимчасово її не виконувати.
- Кешувати певну частину токенів у редіс.
- Відмовитися від даної перевірки, проте забезпечити безпеку іншим способом.

## 5. Ризик втрати accessToken при перезавантаженні сторінки (page reload)

AccessToken зберігається in-memory (Zustand) — при page reload стор очищається, токен губиться. Якщо не реалізована логіка відновлення сесії, юзер буде редіректнутий на `/login` при кожному F5, навіть якщо refresh token у HttpOnly cookie ще дійсний.

**Можливе рішення:**

1. При ініціалізації застосунку (до рендеру захищених роутів) автоматично виконувати `POST /auth/refresh`.
2. Якщо refresh успішний — встановити новий `accessToken` in-memory і дозволити рендер.
3. Якщо refresh неуспішний — редірект на `/login`.
4. На час перевірки показувати loading-стан, щоб уникнути flash of unauthenticated content (FOUC).

## 6. Ризик помилки в RLS policy (витік або блокування даних між тенантами)

RLS policies є критичним шаром безпеки, але їх складно перевіряти вручну. Помилка у policy може призвести до двох сценаріїв: надмірне відкриття (тенант A бачить дані тенанта B) або надмірне блокування (юзер не може отримати власні дані).

**Можливе рішення:**

1. Написати автоматичні інтеграційні тести для RLS перед запуском продакшну:
   - Юзер тенанта A не бачить рядки тенанта B
   - SYSADMIN бачить всі рядки
   - SCHOOL_ADMIN бачить лише рядки свого тенанта
2. Тестувати від імені конкретної DB-ролі через `SET ROLE app_api_role` перед SELECT.
3. Використовувати `EXPLAIN` для перевірки що RLS-фільтр застосовується (шукати `Filter: (tenant_id = current_setting(...))`).
4. Ввести code review checklist для кожної нової таблиці: чи додана RLS policy, чи покрита тестами.

## 7. Ризик викраденого JWT (stolen JWT)

Якщо access token буде викрадено (наприклад, через XSS, витік логів або перехоплення на скомпрометованому пристрої), зловмисник зможе виконувати запити від імені користувача до закінчення строку дії токена.

**Можливе рішення:**

1. Робити короткий TTL для access token (наприклад, 5-15 хв) - є зараз.
2. Refresh token тримати тільки в `HttpOnly` + `Secure` cookie, не в localStorage. - є зараз.
3. Додати ротацію refresh token при кожному `POST /auth/refresh`. - це є.
4. Вести denylist для `jti` і перевіряти його для критичних endpoint-ів. - це є.
5. Логувати аномалії сесій (новий IP/UA, підозріла географія, різкі паралельні запити) і форсовано ревокати сесію. - це є.
6. Мінімізувати дані в JWT (не класти чутливі поля), використовувати тільки потрібні claims. - це є.
