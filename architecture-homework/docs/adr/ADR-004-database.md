## Context

Платформа є мульті-тенантною і має щільно пов'язані сутності (студент учитель, івент - юзер, урок - учень - учитель). Зв'язки типу «один-до-багатьох» і «багато-до-багатьох» потребують підтримки foreign keys та JOIN-запитів.

## Decision

Реляційна база даних PostgreSQL.

## Consequences

### Positive

- Природна модель зв'язків через FK, JOIN
- Вбудований RLS для тенант-ізоляції на рівні БД
- Широка хмарна підтримка: AWS RDS, Aurora PostgreSQL, Google Cloud SQL, Azure, Supabase, Neon
- Сумісність з Prisma ORM, TypeORM та іншими стандартними інструментами
- Managed tier на Supabase або AWS RDS є доступним за ціною

### Negative

- Горизонтальне масштабування запису обмежене: без sharding можемо стикнутися з перформанс проблемою при великих навантаженнях трафіку
- Вертикальне масштабування має фізичну межу (один сервер)
- RLS-policy потребують ретельного дослідження: помилка в policy може призвести до надмірного відкриття або блокування доступу до рядків + я це ніколи не робила, це відповідальна частина, і необхідно буде велику частину часу придліитина це

## Alternatives Considered

### MongoDB (Document Store)

**Розглядалось**: гнучка схема, горизонтальне шардування «з коробки», зручно для ієрархічних документів.  
**Відхилено**: відсутній вбудований RLS — тенант-ізоляцію треба реалізовувати виключно на рівні застосунку, що збільшує ризик витоку. Транзакції між колекціями складніші й повільніші. Зв'язки між сутностями вимагають lookup-запитів, що є повільнішими або дублікації даних.

### MySQL / MariaDB

**Розглядалось**: популярний варіант із широкою підтримкою.  
**Відхилено**: MySQL не має повноцінного RLS.

### Database-per-Tenant (один PostgreSQL на кожного тенанта)

**Розглядалось**: максимальна ізоляція — кожен тенант має власну БД.  
**Відхилено**: надмірні операційні витрати (міграції окремо для кожного тенанта), складність cross-tenant запитів (необхідних для глобальних подій), idle-витрати на кожну пусту базу. Суперечить бюджетним і часовим обмеженням (ADR-002).
