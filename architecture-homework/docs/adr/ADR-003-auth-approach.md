## Context

Платформа вимагає захищеного входу, адже містить особисті дані вчителів, а також приватні репорти школи..

## Decision

Авторизація через magic link + JWT access token + refresh token in cookie

## Consequences

### Positive

- посилений захист через двофакторну авторизацію - недостатньо просто вгадати пароль, треба ще мати доступ до пошти
- швидкоплинний термін дії ексіс токена, а також його зберігання in-memory зменшує ризик його використання зловмисниками - його складніше отримати ніж якщо б він зберігався у локал сторіджі, а також при отриманні термін дії швидко скінчиться
- зберігання ревреш токену як кукі, що не можна отримати через браузер і який можна лише з поточного домена відправити зменшує ризик крадіжки

### Negative

- ризик крадіжки ексіс токена лишається
- складність ротації токена якщо крадіжка зафіксована - треба вводити денайліст і зберігати токен gti у базі даних
- при доступу до пошти + занючи пароль, зловмисник зможе залогінитися

## Alternatives Considered

### 1. Логін/пароль + JWT (без magic link)

Стандартний підхід: користувач вводить email і пароль, сервер повертає токени.

**Відхилено**, бо платформа містить особисті дані школярів та вчителів — слабкі або повторно використані паролі є реальним ризиком.

### 2. OAuth 2.0

Делегування автентифікації стороннім провайдерам.

**Не розглядалося як основний варіант**, бо частина шкіл може не мати корпоративних акаунтів Google Workspace або Microsoft 365, а залежність від зовнішнього провайдера ускладнює onboarding нових тенантів. Може бути додано пізніше як опціональний метод входу поряд із magic link.

### 3. Лише access token (без refresh token)

Довгоживучий JWT без механізму оновлення.

**Відхилено** — довгий TTL токена суперечить вимозі мінімізувати вікно компрометації.
