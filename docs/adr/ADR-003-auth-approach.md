## Context

Платформа вимагає захищеного входу, адже містить приватні дані школи.

## Decision

Авторизація через magic link + JWT access token + refresh token у cookie

## Consequences

### Positive

- посилений захист через двофакторну авторизацію - недостатньо просто вгадати пароль, треба ще мати доступ до пошти
- швидкоплинний термін дії ексіс токена, а також його зберігання in-memory зменшує ризик його використання зловмисниками через XXS
- зберігання ревреш токену як кукі, що не можна отримати через JS і який відправляється лише з поточного домену

### Negative

- деякий ризик крадіжки ексіс токена лишається
- складність ротації токена якщо крадіжка зафіксована - треба вводити денайліст і зберігати токен gti у базі даних, а потім при окжному запиті до найважливіших ендпоїнтів перевіряти денайліст
- при доступу до пошти + знаючи пароль, зловмисник зможе залогінитися

## Alternatives Considered

### 1. Логін/пароль + JWT (без magic link)

Стандартний підхід: користувач вводить email і пароль, сервер повертає токени.

**Відхилено**, бо платформа містить особисту інформацію школи, іноді пароль можна вгадати.

### 2. OAuth 2.0

Делегування автентифікації стороннім провайдерам.

**Можна розглянути як додатковий варіант**

### 3. Лише access token (без refresh token)

Довгоживучий JWT без механізму оновлення.

**Відхилено** — довгий TTL токена дозволяє його безмежне використання після крадіжки.
